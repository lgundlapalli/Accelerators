## Design Decisions

### Why Solution Approach for Path A?

**Rationale:** Path A PRDs (no prototype) lacked solution description. Without it, engineering gets problem + metrics but no concept of what to build.

**Path B has Prototype Evidence section** which documents:
- What was built (3 prototypes)
- Which won and why
- What was validated/invalidated

**Path A needed equivalent** documenting solution rationale:
- What we're building (concept)
- Why this approach (rationale)
- Key components (high-level)
- Alternatives considered and rejected

**Alternative considered:** Leave Path A as problem-only document (product brief) — rejected because v1.3 already asks for rollout/dependencies/risks which require solution context.

### Why Input → Expected over GOOD/BAD/REJECT?

**Rationale:** GOOD/BAD/REJECT is training-oriented. Engineering needs specification: "Given input X, system does Y."

**What changed:**
- Primary format: Input → Expected with edge case flag
- GOOD/BAD/REJECT: Optional supplement for training purposes

**Why keep GOOD/BAD/REJECT option:** Some teams use PRDs to train models. Keeping it as supplement allows both use cases.

### Why Simplify Rollout from A/B Experiment Details?

**Rationale:** MDE, statistical power, randomization unit are experiment design details that belong in technical spec, not PRD.

**PRD-level rollout should answer:**
- When? (Q2/Q3 timeline)
- How much? (10% → 50% → 100% phasing)
- What gates? (metrics thresholds, approvals)
- Kill switch? (who, how fast)

**Detailed experiment design answers:** MDE calculation, power analysis, randomization mechanics, sample size → technical spec phase.

### Why AI Feature Readiness as Separate Subsection?

**Rationale:** AI features fail when data/systems/infrastructure aren't ready. Making this explicit prevents mid-build blockers.

**What it forces:**
- Data audit: What data does AI need? Does it exist? Is it clean?
- Integration check: What systems must AI connect to? Are APIs ready?
- Infrastructure check: Where does model run? Is it provisioned?

**Alternative considered:** Merge into Dependencies list — rejected because AI readiness is multi-dimensional (data + systems + infrastructure) and benefits from structured subsection.

### Why Table Formats for Dependencies/Guardrails/Risks?

**Rationale:** Scannability. PRDs are decision documents. Tables allow quick scanning of key information.

**Readability test:** Can a reader scan Dependencies table and immediately see owners and status? Yes. Can they scan prose paragraphs? No.

**Applies to:** Dependencies, Guardrail Metrics, Top Risks — all 3 are structured data best presented as tables.

### Why Make Post-Ship Universal?

**Rationale:** All PRDs should be living documents, not just prototype-validated ones.

**What Post-Ship captures:**
- Results link
- What surprised us (user behavior, data insights)
- What will we change (iteration decisions)
- Decision (iterate/scale/retire)

**Path A PRDs need this too:** Even non-prototyped features have post-launch learnings worth documenting.

## Lessons Learned

### What Worked Well

1. **TDD-inspired approach validated fixes** — Building reference PRDs first proved target quality was achievable before updating skill
2. **Gap analysis from evaluator** — Developing evaluator first revealed exactly what v1.3 was missing
3. **Two test cases covered both paths** — TC1 (Path B, AI) and TC2 (Path A, non-AI) validated all 10 gaps
4. **Table formats improve scannability** — Dependencies, Guardrails, Risks all benefit from tabular structure
5. **Solution Approach critical for Path A** — Without it, Path A PRDs were incomplete (problem without solution)

### What We'd Do Differently

1. **Build evaluator first** — We accidentally did this (built evaluator before improving writer) which was correct approach. In future, always build evaluation framework before building generator.
2. **More test cases** — Could test edge cases (AI + Path A hypothetical, multi-phase rollout complexity)
3. **Automated validation** — Could script evaluator to run on every PRD generated, showing score immediately

### Open Questions

1. **Should Solution Approach be optional for Path A if user has prototype-like data?** — Currently required for all Path A, but some Path A features may have lightweight prototypes
2. **Should table formats be enforced by skill or suggested?** — Currently suggested but not strictly enforced in output validation
3. **Should MDE/statistical power be completely removed or optional?** — Currently removed, but some teams may want it in PRD vs separate spec
